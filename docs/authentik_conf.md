# Контекст проекта и техническое задание для настройки Authentik (GitOps-подход)

## 1. Общий контекст системы

Мы разрабатываем **Zero-Dashboard платформу управления репутацией** для микро- и малого бизнеса.

- **Основной интерфейс взаимодействия:** SMS-вебхуки (Twilio, Sinch) и Telegram Bot API. Традиционной панели управления (админки) для клиентов нет.
- **Технологический стек:**
  - API-шлюз и бизнес-логика: **Go** (I/O-bound задачи, высококонкурентный прием вебхуков).
  - Анализ текста и парсинг (GMB, Yelp, TripAdvisor): **Rust** (CPU-bound задачи).
  - Обратный прокси и API Gateway: **Traefik v3**.
  - Шина сообщений: **NATS Core / JetStream**.
  - Телеметрия: **OpenTelemetry (OTel) Collector, Jaeger, Prometheus**.
  - Среда развертывания: **Docker Compose** (изолированная сеть `reputation-tier`).

## 2. Архитектурная цель для Authentik

Необходимо полностью реализовать **GitOps-подход** для управления доступом. Вся конфигурация Authentik (потоки, этапы, политики, сервисные аккаунты) должна быть описана декларативно в виде **Authentik Blueprints (YAML)**. Взаимодействие с UI админки должно быть сведено к нулю. Потоки авторизации должны быть оптимизированы под обслуживание автоматических API-запросов и вебхуков, проходящих через Traefik Forward Auth.

## ИНСТРУКЦИЯ ДЛЯ ИИ-ИСПОЛНИТЕЛЯ

Твоя задача — последовательно выполнить следующие инженерные задачи. Генерируй только чистый, валидный код и декларативные конфигурации, оптимизированные под production-нагрузки.

### ЗАДАЧА 1: Оптимизированный Docker Compose манифест

**Цель:** Создать конфигурацию инфраструктуры Authentik, полностью готовую к горизонтальному масштабированию.
**Требования:**

1. Разделить компоненты: Authentik Core, Authentik Worker, PostgreSQL (с постоянным volume), Redis (как кэш сессий и хранилище стейта политик).
2. Настроить лимиты ресурсов (CPU/RAM) для контейнеров, чтобы избежать OOM-killer под пиковой нагрузкой вебхуков.
3. Добавить `healthcheck` для каждого сервиса, гарантирующий правильный порядок запуска (Postgres -> Redis -> Core -> Worker).
4. Настроить автоматическое монтирование директории с локальными Blueprints (`/blueprints/`) для их автоматического применения при старте.

---

### ЗАДАЧА 2: Декларативный Authentik Blueprint (YAML)

**Цель:** Сгенерировать единый Blueprint-файл (версии 1) для инициализации структуры доступа платформы.
**Требования:**

1. **Потоки (Flows):** Создать кастомный поток аутентификации API-запросов (`api-validation-flow`), который пропускает запросы без вывода HTML-форм (Explicit API Auth).
2. **Сервисные аккаунты (Service Accounts):** Описать создание учетных записей для Go-микросервисов (`sms-gateway-service`) и Rust-воркеров с генерацией долгоживущих API-токенов.
3. **Провайдеры и приложения:** Описать `Proxy Provider` в режиме `Forward Auth (Single Application)`, привязанный к `api-validation-flow`.
4. **Маппинг атрибутов:** Настроить передачу кастомных метаданных организации в Downstream-сервисы через HTTP-заголовки.

---

### ЗАДАЧА 3: Expression Policies (Политики на базе выражений)

**Цель:** Написать производительные и безопасные политики авторизации на встроенном синтаксисе Authentik (Python-like).
**Требования:**

1. **Политика валидации Twilio/Sinch:** Написать выражение, которое проверяет IP-адрес источника входящего запроса. Если запрос идет не из разрешенного CIDR-пула провайдера связи, доступ блокируется (HTTP 403).
2. **Политика обогащения контекста (Tenant ID extraction):** Написать выражение, которое парсит метаданные токена/сервисного аккаунта, извлекает идентификатор B2B-клиента (`tenant_id`) и внедряет его в контекст авторизации Authentik, чтобы Traefik мог пробросить его как заголовок `X-Tenant-ID`.

---

### ЗАДАЧА 4: Конфигурация Traefik v3 Middleware (Labels)

**Цель:** Написать конфигурацию для Traefik, связывающую обратный прокси с Authentik Outpost через Forward Auth.
**Требования:**

1. Использовать Docker labels для декларативного описания middleware.
2. Трафик между Traefik и Authentik Outpost должен идти по HTTP (порт 9000) внутри Docker-сети `reputation-tier`. Исключить самоподписанные SSL-сертификаты на внутреннем участке.
3. Настроить `authResponseHeaders` для проброса заголовков: `X-Authentik-Username`, `X-Authentik-Jwt`, `X-Tenant-ID`.
4. Написать регулярное выражение (`skip-auth-regex`) для обхода авторизации при запросах к метрикам `/metrics` и хелсчекам `/healthz` внутренних сервисов.

---

### ЗАДАЧА 5: Наблюдаемость и OpenTelemetry

**Цель:** Обеспечить интеграцию Authentik в общую систему мониторинга.
**Требования:**

1. Предоставить конфигурацию для сбора Prometheus-метрик с инстансов Authentik Core и Worker.
2. Настроить проброс заголовка трассировки W3C Trace Context (`traceparent`) через Authentik, чтобы задержки на этапе авторизации корректно сопоставлялись со спанами в Jaeger.
