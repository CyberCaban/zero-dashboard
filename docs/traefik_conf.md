# Архитектурный манифест: Цели и принципы конфигурации Traefik API Gateway

**Проект:** Платформа управления репутацией (Zero-Dashboard)  
**Роль компонента:** Единая точка входа (API Gateway / Edge Proxy)  
**Стек окружения:** Docker Compose, Go (Gateways), Rust (Analyzers), NATS, Zitadel/Authentik (AAA)

---

## 1. Введение и бизнес-контекст

В архитектуре **Zero-Dashboard**, где взаимодействие с клиентами и мерчантами происходит преимущественно через асинхронные каналы (Telegram, WhatsApp, SMS), `Traefik` выполняет роль критически важного «цифрового пограничника».

Поскольку система не имеет классической тяжелой административной панели, её периметр состоит из множества разрозненных вебхуков и служебных эндпоинтов. Конфигурация Traefik должна превратить эти изолированные микросервисы в монолитный, защищенный и отказоустойчивый API-контур.

---

## 2. Ключевые архитектурные цели

При проектировании и написании конфигурации Traefik (через динамические `labels` в Docker и статические `traefik.yml` файлы) мы преследуем 4 главные цели:

### Цель №1: Нулевое обслуживание (Zero-Maintenance & Service Discovery)

Конфигурация не должна быть статичной. Мы исключаем ручное прописывание IP-адресов или портов контейнеров.

- **Реализация:** Использование провайдера `providers.docker`. Traefik должен автоматически слушать Docker-демона.
- **Результат:** При масштабировании Go-шлюзов (`docker compose up --scale messenger-gateway=3`) Traefik обязан самостоятельно обнаружить новые инстансы и включить внутреннее балансирование нагрузки между ними без перезапуска прокси.

### Цель №2: Абсолютная автоматизация криптографии (Авто-SSL)

Безопасность персональных данных клиентов (номера телефонов, тексты отзывов) требует повсеместного шифрования. При этом команда разработки не должна тратить время на менеджмент сертификатов.

- **Реализация:** Интеграция с Let's Encrypt через ACME-клиент (`certresolver`) на уровне конфигурации точек входа (Entrypoints).
- **Результат:** Автоматический выпуск, привязка и обновление SSL-сертификатов для всех субдоменов (`api.*`, `auth.*`). Весь входящий трафик принудительно редиректится с `HTTP (80)` на `HTTPS (443)`.

### Цель №3: Защита бэкенда от деградации (Resiliency & Rate Limiting)

Публичные эндпоинты вебхуков мессенджеров подвержены рискам спам-атак или лавинообразного роста трафика в часы пик (например, пятница вечер в ресторанах).

- **Реализация:** Внедрение слоев промежуточного ПО (`Middlewares`) для каждого критического маршрута.
- **Результат:** Ограничение частоты запросов (Rate Limiting) и лимитирование одновременно открытых соединений (In-Flight Requests). Попытки заспамить систему отсекаются на уровне прокси, возвращая злоумышленнику `HTTP 429 Too Many Requests`, не нагружая при этом брокер NATS и бэкенд на Go.

### Цель №4: Изоляция контуров безопасности (Сегментация трафика)

Разделение входящего потока данных на "безопасный" (требующий авторизации AAA) и "открытый" (вебхуки от официальных платформ).

- **Реализация:** Использование плагина `ForwardAuth` для связи маршрутизатора Traefik с identity-платформой (Zitadel/Authentik).
- **Результат:** Микросервисы за прокси-сервером полностью освобождаются от кода проверки сессий и токенов. Они получают от Traefik уже «очищенный» трафик с гарантированными заголовками авторизации.

## 3. Логическая структура маршрутизации (Routers & Middlewares)

Конфигурация должна четко делить трафик на три независимых контура, к каждому из которых применяется свой набор правил:

### Контур А: Публичные Вебхуки (`Open Webhooks`)

- **Маршрут:** `api.domain.com/v1/webhooks/>`
- **Цель:** Прием данных от Telegram API, Twilio, Sinch.
- **Правило безопасности:** Доступ открыт для всего мира. Авторизация запроса происходит на уровне самого Go-шлюза по уникальному хэшу в URL.
- **Обязательные Middlewares:** Стрикт-лимитер частоты запросов (например, не более 100 rps с одного IP), ограничение размера тела запроса (MaxRequestBodyBytes = 2MB для защиты от загрузки тяжелых медиафайлов).

### Контур Б: Интеграции и AAA (`Identity Gateway`)

- **Маршрут:** `auth.domain.com/>`
- **Цель:** Авторизация владельцев бизнеса, связывание их аккаунтов Google/Яндекс с платформой.
- **Правило безопасности:** Трафик пробрасывается напрямую в контейнер `Zitadel / Authentik`.
- **Обязательные Middlewares:** Настройка заголовков безопасности (Secure Headers: STS, X-Frame-Options, CSP).

### Контур В: Служебный и Партнерский API (`Secure API`)

- **Маршрут:** `api.domain.com/v1/secure/>`
- **Цель:** Запросы от PWA-форм (микро-лендинги отзывов), партнерские интеграции.
- **Правило безопасности:** Доступ строго после проверки токена.
- **Обязательные Middlewares:** `ForwardAuth`. Traefik перенаправляет заголовки `Authorization: Bearer` в систему AAA. Только при ответе `HTTP 200` от AAA запрос идет в бизнес-логику Go.

## 4. Метрики успешности (Definition of Done для конфигурации)

Конфигурация Traefik считается успешно написанной и готовой к продакшену, если выполняются следующие критерии:

> 1. **SSL Labs Grade А+:** При тестировании домена утилитами проверки SSL система получает наивысшую оценку безопасности TLS.
> 2. **Контейнеры «не знают» про SSL:** В кодовой базе Go и Rust сервисов отсутствуют пути к файлам `.crt` и `.key`. Сервисы слушают порт `80` и работают по чистому HTTP внутри сети Docker.
> 3. **Логирование в формате JSON:** Все access-логи Traefik структурированы в JSON для последующей бесшовной интеграции с Vector / Loki / Vector.
> 4. **Проброс реального IP:** Внутренние сервисы видят реальный IP-адрес клиента (через заголовок `X-Real-IP`), а не внутренний IP-адрес прокси-сервера. Это критично для аналитики локации отзывов.
