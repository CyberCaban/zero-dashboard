### Фаза 1: Инфраструктурный фундамент и Слой транспорта (Скелет)

На этом этапе код сервисов не пишется. Задача — подготовить «рельсы», по которым будут ходить данные.

1. [x] **Развертывание локальной шины:** Написание базового Docker Compose. Запуск `NATS` с включенным JetStream, `Redis` и `PostgreSQL`.
2. [x] **Определение контрактов данных:** Описание структур сообщений (DTO). Поскольку у вас стек Go и Rust, идеальным выбором будет **Protobuf**, но для старта подойдет и строгий, задокументированный JSON-схема контракт.
3. [x] **Фиксация структуры субъектов (Subjects) в NATS:** Четкое проектирование дерева топиков с учетом мультитенантности (например, `reputation.tenant.<tenant_id>.<service>.<action>`).

### Фаза 2: Проверка концепции асинхронного ввода-вывода (Первый запуск)

Цель — проверить паттерн Request-Reply и скорость прохождения сообщений в Go-стеке.

1. [x] **Разработка минимального `Integration Gateway` (Go):** Он просто принимает сырой HTTP-запрос (без проверки подписей и Zitadel) и бросает JSON в NATS.
2. [x] **Разработка минимального `Review Gating Service` (Go):** Он слушает NATS, принимает сообщение, имитирует работу конечного автомата (FSM) и по паттерну Request-Reply (через `_INBOX`) мгновенно возвращает ответ в гейтвей.
3. [x] **Результат фазы:** У вас есть сквозной проход трафика `HTTP -> Go Gateway -> NATS -> Go Service -> NATS -> Gateway -> HTTP`.

### Фаза 3: Периметр безопасности и Изоляция тенантов (Identity & AAA)

Теперь, когда данные циркулируют, нужно закрыть систему от внешнего мира и связать сущности.

1. **Настройка Прокси:** Подключение `Traefik` или `Nginx` перед вашим Go-гейтвеем.
2. **Инициализация IAM (Zitadel / Authentik):** Описание конфигурации в YAML (GitOps). Создание первого тестового B2B-клиента (`org_123`) и сервисных аккаунтов для микросервисов.
3. **Внедрение Forward Auth / OAuth2-Proxy:** Настройка прокси так, чтобы он не пускал запросы к гейтвею без валидации в IAM.
4. **Реализация Token Propagation (Проброс контекста):** Доработка `Integration Gateway` (Go). Теперь он считывает заголовки от прокси, сопоставляет их, достает `tenant_id` и **инжектирует его в заголовки сообщения NATS** (`msg.Header.Set("X-Tenant-ID")`).

### Фаза 4: Подключение тяжелых вычислений (Rust Data Plane)

Только теперь, когда транспорт безопасен и изолирован, можно внедрять сложную логику.

1. **Разработка `Sentiment & Topic Analyzer` (Rust):** Сервис подключается к NATS, забирает сообщения из Queue Group. Он парсит текст, извлекает `X-Tenant-ID` из заголовков NATS и понимает, в контексте какого клиента он считает тональность.
2. **Разработка `Multi-Platform Scraper` (Rust):** Реализация логики парсинга. Подключение `Browserless` в Docker Compose, настройка BrowserQL мутаций для отсечения медиа-контента, внедрение ротации прокси. Скрейпер начинает поставлять сырые данные в топики NATS, откуда их забирает `Review Gating Service`.

### Фаза 5: Распределенная телеметрия (Observability)

Система работает, но она «слепая». Нужно подтвердить математическую модель задержек $L_{\text{total}}$.

1. **Развертывание стека мониторинга:** Добавление в Docker Compose `OpenTelemetry Collector`, `Jaeger` и `Prometheus`.
2. **Инжекция W3C Trace Context:** \* В `Integration Gateway` (Go) пишется генерация Root Span (`traceparent`).

- В заголовки NATS добавляются метаданные трассировки.
- В Rust-сервисах настраивается маппинг контекста внутри асинхронного рантайма `Tokio` (через `Context::attach`).

3. **Результат фазы:** В интерфейсе Jaeger вы видите полный визуальный граф: сколько микросекунд вебхук шел через Traefik, сколько заняла авторизация, сколько сообщение лежало в очереди NATS и сколько миллисекунд Rust-сервис тратил на токенизацию без пауз GC.

### Фаза 6: Тюнинг под нагрузки и Продакшн (Hardening)

Оптимизация системы для защиты от деградации памяти и блокировок связи.

1. **Ограничение JetStream:** Включение Ingest Rate Limiting (`max_buffered_size`), настройка TTL для SMS (чтобы клиенты не получали ночной спам из-за задержек оператора).
2. **Rate Limiting на отправку:** Группировка воркеров `Push Notification Service` в единую `NATS Queue Group`, чтобы Round-Robin алгоритм четко дозировал QPS в сторону Twilio/Sinch.
3. **Оптимизация Docker:** Фиксация production-тегов образов, жесткое ограничение лимитов памяти (`mem_limit`) для Go-сервисов (контроль GC) и выделение ресурсов для JVM-компонентов (если во втором варианте архитектуры используется Kafka).

### Архитектурный совет

Не переходите к **Фазе 4 (Rust)**, пока у вас полностью не заработает **Фаза 3 (Безопасность и NATS-заголовки)**. Самая частая ошибка — написать сложнейшие алгоритмы парсинга и анализа текста, а потом пытаться «вкрутить» туда контекст организации и безопасность, что приводит к переписыванию половины кодовой базы.
